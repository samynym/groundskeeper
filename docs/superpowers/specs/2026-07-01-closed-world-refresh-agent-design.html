Design Spec · Brainstorm output

Groundskeeper A closed-world content refresh agent — maintains an existing site without ever asserting a fact it can’t cite.

● Design approved · pending spec review 2026-07-01 Working name: groundskeeper First target: Steady · growsteady.me
The one rule the whole product hangs on

A claim that can’t cite a source in the evidence base does not ship.

Every competitor runs an open-world writer that can hallucinate, with the human as the safety net. Groundskeeper inverts it: the agent runs closed-world, and the guard — not the human — is the safety net. The human is only the merge gate.

Why this exists

The open seam between the incumbents
Outrank

Sprays new posts

Autonomous but fire-and-forget. Never maintains what it published. Targets non-marketers.

Frase

Not autonomous

Has maintenance + AEO features, but human drives a dashboard. Built for marketing teams.

Slate

Enterprise-only

Genuinely autonomous closed-loop — but 1,000+ pages, $10–25k/yr. Abandons the bottom of the market.

The gap →

An autonomous maintenance loop for the non-marketer / small site — below Slate’s floor — made safe by closed-world grounding.

Steady is the ideal first target: its evidence base is already structured and cited, and it’s a YMYL site where grounding matters most.

Scope

Goals & non-goals

✅ Goals

  • Prove an agent can maintain a real YMYL site unattended without producing garbage.
  • Refresh existing pages, driven by underperformance + staleness.
  • Ground every edit strictly in a curated, cited evidence base.
  • Ship review-ready PRs; human approves the merge.
  • Dogfood on Steady; keep clean seams to productize later.

🚫 Non-goals (v1)

  • New-page generation.
  • GEO / AI-citation tracking.
  • Web dashboard, auth, multi-tenancy.
  • CMS adapters (WordPress/Webflow).
  • Auto-merge — human approves every PR.
  • Deciding vertical vs horizontal productization.

How it’s wired

Architecture — a separate repo that operates on a checkout

Groundskeeper is its own repository. It never stores the target site’s content. It operates on a local checkout of the target repo at a configured path:

  • Reads content + citations via a ContentSource adapter.
  • Reads performance via the target’s existing GSC tooling (scripts/seo-loop.mjs).
  • Drafts edits → grounding guard → verifies the target’s build.
  • Branches, commits, pushes, opens a PR on the target repo via gh.

The Steady adapter is the first and only ContentSource for the MVP — it encodes Steady’s lib/benchmarks/* layout. A second site later = a second adapter, nothing else. The separate repo forces that boundary to be real from day one.

One cycle

Data flow, start to PR
  1. 1PerformanceSignal GSC snapshot for sc-domain:growsteady.me.
  2. 2TargetSelector Rank all procedure/week pages by underperformance (page 2–3 position; high-impression / low-CTR) + staleness (basis:"interpolated", thin sources[], long time since last git edit). Pick the top one.
  3. 3EvidenceRetriever Assemble the citable fact set for that target.
  4. 4RefreshDrafter Propose a concrete diff: prose tightening, FAQ/intent fixes, promoting an interpolated value to measured when the dossier already documents it (never fetching a new source), metadata / internal-link fixes for CTR.
  5. 5GroundingGuard Validate every claim against the evidence set. Unsupported → strip or reject; log why.
  6. 6BuildVerifier Typecheck + lint + test the edited checkout. Fail → abort, no PR.
  7. 7PRWriter Open a PR whose body lists each change with the exact source it traces to.
  8. 8Human Review the PR. Merge → ships via Vercel SSG.

The crux

Grounding guard — two layers, a claim must pass both

This runs on the agent’s own output, before any human sees it. It’s what lets the agent run unattended.

Layer 1 · Mechanical

Deterministic & cheap — leverages the fact that citations are structured objects.

  • Every numeric value carries a basis + resolvable sourceUrl.
  • Every prose claim references an existing sources[] entry.
  • No sourceUrl may be invented — must already exist.
  • measured never replaced by a worse interpolated.
  • The measured/interpolated honesty flag preserved, never silently downgraded.

Layer 2 · Adversarial judge

A separate LLM pass, for prose.

  • Reads each new sentence: “is this supported by the cited passage — yes/no?”
  • Defaults to reject when uncertain.
  • Only claims surviving both layers ship.
Design stance: false negatives — a fabricated claim that slips through — are the only unacceptable failure. The guard is tuned to over-reject. A rejected-but-true claim just doesn’t get made this cycle. That’s fine.

Who pulls the trigger

Autonomy model
v1

Autonomous up to the PR

Human approves the merge. No auto-merge on a YMYL site.

Later

Selective auto-merge

Only low-risk classes (metadata, internal links), once the guard’s reject rate + human accept rate are proven. Not v1.

Definition of done

MVP success = all three
1 · Zero ungrounded claims

Every claim in every PR traces to a real source (manual audit confirms the guard). Pass/fail bar — nothing else matters if this fails.

2 · Useful, not noise

You’d merge ≥50% of PRs with at most light edits.

3 · Unattended

A full cycle runs end-to-end on a schedule with no hand-holding.

Traffic lift is the eventual proof but too slow to gate the MVP — it’s a post-MVP metric.

Deliberately not building

Out of scope (YAGNI)
Dashboard / web UI CMS adapters New-page generation GEO / AI-citation tracking Multi-tenant / auth Auto-merge Productization model

The ContentSource and GroundingGuard seams make these cheap to add later. We just don’t build them now.

Proving it works

Testing strategy

Guard unit tests

Golden set mixing grounded + subtly fabricated claims. Must catch 100% of fabrications — false negatives fail the build.

Dry-run mode

Produce the diff + audit table without opening a PR. Fast iteration.

One real E2E PR

A single end-to-end PR on Steady as the acceptance test.

Left for later
